1. Collision Clarity — Hard Edges

Design Priority

Problem: The rotating dashed ring around the balls is visually prominent but does NOT count as collision. Players are confused about when contact actually occurs — they think the ring touches first.

Current (Confusing)

Dashed ring rotates visually outside the collision radius, making it unclear where the actual edge is.

Proposed (Clear)

Solid hard edge with subtle outline. Optional faint glow stays within collision boundary. Exact contact point is obvious.

Recommended Changes

  • ✗ Remove the rotating dashed ring entirely (simplest solution) — it's decorative and causes confusion
  • ✓ Add a solid 2px stroke (white/light) around the ball to mark the exact collision boundary
  • ✓ If you want movement flair, use a subtle inner glow or gradient that does NOT extend beyond the collision radius
  • ✓ On tag collision, flash both balls with a bright 200ms highlight to confirm contact occurred

Implementation Notes

In src/game/renderer.ts (or GameCanvas.tsx draw function):

// Remove dashed ring drawing
// Add solid edge:
ctx.beginPath()
ctx.arc(ball.x, ball.y, ball.radius, 0, Math.PI * 2)
ctx.strokeStyle = 'rgba(255, 255, 255, 0.6)'
ctx.lineWidth = 2
ctx.stroke()

// Optional: subtle inner gradient fill
const gradient = ctx.createRadialGradient(
  ball.x, ball.y, ball.radius * 0.3,
  ball.x, ball.y, ball.radius
)
gradient.addColorStop(0, ball.color)
gradient.addColorStop(1, darken(ball.color, 0.3))
ctx.fillStyle = gradient
ctx.fill()

The collision detection already uses the exact ball.radius, so no physics changes needed.

2. Gravity Powerup — Balance & Visual Feedback

Design Priority

Problem: Gravity Well pulls the Hider toward the Seeker too strongly, making it impossible to escape once activated. The suction feels overpowered and reduces gameplay variety.

GRAVITY_PULL
Current strength constant
0.003
Sample value (to be reduced ~40% → 0.0018)
Distance-based
Proposed: stronger when close, weaker when far

Balance Proposal

  • ✗ Constant pull of ~0.003 per frame is too high — Hider gets uncontrollably sucked in
  • ✓ Reduce base pull strength by 40-50% to ~0.0018
  • ✓ Make it distance-based: pull ∝ 1/distance (or clamped) so it's weaker at range, stronger up close
  • ✓ Add subtle visual indicator on Seeker: pulsing gravity ring effect to show active field
  • ✓ Add subtle HUD indicator for Hider when under gravity effect (small icon + timer)

Visual Feedback Ideas

Seeker with Gravity

Pulsing concentric rings radiating outward

Hider Affected

Small gravity icon in HUD + pull arrow toward Seeker

Implementation

In src/constants.ts:

export const GRAVITY_PULL_BASE = 0.0018  // reduced from 0.003
export const GRAVITY_PULL_MAX = 0.004
export const GRAVITY_PULL_MIN_DIST = 100  // min distance clamp
export const GRAVITY_PULL_MAX_DIST = 500  // max distance (falls off beyond)

In src/game/powerups.ts gravity activation:

const distance = Math.hypot(hider.x - seeker.x, hider.y - seeker.y)
const clampedDist = Math.max(GRAVITY_PULL_MIN_DIST,
                    Math.min(distance, GRAVITY_PULL_MAX_DIST))
const pullStrength = GRAVITY_PULL_BASE * (GRAVITY_PULL_MAX_DIST / clampedDist)

// Apply pull vector toward seeker
hider.vx += (seeker.x - hider.x) / distance * pullStrength
hider.vy += (seeker.y - hider.y) / distance * pullStrength

3. Replay Flow — Survival Mode Consistency

Design UX

Problem: After finishing a Survival round, clicking "Replay" incorrectly starts a new Vs. round instead of continuing Survival. The flow breaks player expectations.

Current Flow Issues

  • Survival mode: single round, Hider vs CPU Seeker, survive as long as possible
  • Round ends → "Replay" button incorrectly triggers startNewRound() which goes to versus rotation
  • Player expects: same Survival setup, new round, keep trying to beat personal best

Proposed Solution

  • ✓ In MatchOverlay.tsx, detect current matchConfig.gameMode
  • ✓ If mode is 'survival', Replay button should call restartSurvival() (new round, same mode, no role swap)
  • ✓ If mode is 'versus', retain current behavior (next round with role swap)
  • ✓ Update button label: "Play Again" → "Try Again" for Survival to be clearer

State Machine Impact

The App state machine needs a survival_restart transition from match_over back to round_intro with survival flags preserved.

// In App.tsx
const handleReplay = () => {
  if (matchConfig.gameMode === 'survival') {
    // Survival: restart fresh, Hider always player, no role swap
    setGamePhase('round_intro')
    resetRound({ isSurvival: true, keepConfig: true })
  } else {
    // Versus: normal progression
    if (roundsRemaining > 0) {
      setGamePhase('round_intro')
      swapRoles()
    } else {
      // match over, go to menu or start new match
    }
  }
}

Summary & Order of Work

Three design-focused tasks. Recommended implementation order:

  1. Gravity balance — single constant change + optional visual feedback (fast win)
  2. Collision clarity — renderer tweak, removes confusing element (low risk)
  3. Replay flow — state machine change, needs careful testing (medium complexity)

All three are relatively isolated. No major refactors needed.
